# ADR-0001: Evolution Strategy — DGM-H 방식 vs 증분 개선 vs 하이브리드

- Status: accepted
- Date: 2026-04-13
- Related docs: ../PRD.md, ../ARCHITECTURE.md

## Context

HyperAgents 논문(DGM-H)은 완전한 진화 루프를 제안한다: N개 변종 생성 → 각 변종 평가 → 아카이브에 최적 보관 → 부모 선택 → 반복. 이 접근법은 탐색 공간이 넓고 최적해가 불분명한 상황에서 효과적이다.

그러나 현재 시스템은 **1인 개발자의 개인 도구**다. 전체 진화 루프를 돌리면 에이전트 프로파일 하나를 개선하는 데 N번의 LLM 호출 + N번의 평가가 필요하다. 에이전트 13개, 스킬 17개를 모두 진화시키면 비용이 폭발한다.

반면 증분 개선(세션 분석 → 단일 개선안 제안 → 적용 → 다음 세션에서 효과 확인)은 비용이 낮지만, 지역 최적(local optimum)에 갇힐 위험이 있다. 특히 에이전트 구조 자체를 재설계하거나 새로운 에이전트/스킬을 생성하는 등 큰 변화에는 부적합하다.

핵심 질문: **일상적 개선과 구조적 변화를 어떻게 균형잡을 것인가?**

### 추가 컨텍스트

- 현재 시스템의 agent-registry는 symlink 기반이므로 instructions.md 수정이 즉시 반영됨 (spike-2 확인)
- 세션 JSONL에서 성능 신호 추출이 가능함 (spike-3 확인)
- 기존 parallel-codex 스킬이 worktree 기반 병렬 실행을 지원하므로 다변종 생성 인프라가 이미 존재함
- workflow.toml의 `max_helpers_per_fanout = 4`가 현재 팬아웃 상한

## Options Considered

### Option A: Full DGM-H 진화 루프

매 개선 사이클마다 N개(예: 4개) 변종을 생성하고, 각각을 평가(시뮬레이션 세션 또는 실제 적용 후 측정)한 뒤 최적 변종을 선택한다.

**장점:**
- 탐색 공간을 넓게 커버하여 전역 최적에 가까운 해를 찾을 수 있음
- HyperAgents 논문에서 입증된 수렴 속도 (7세대 만에 기존 대비 ~25% 개선)
- 변종 간 비교를 통해 어떤 변화가 효과적인지 직접 학습 가능
- 아카이브 메커니즘으로 다양성 유지 — 과적합 방지

**단점:**
- 비용 문제: 변종 4개 × 에이전트 13개 = 사이클당 최소 52회 LLM 호출 (평가 별도)
- 평가 난이도: 에이전트/스킬의 "품질"을 자동 평가하려면 별도 평가 기준과 벤치마크 필요
- 과잉 엔지니어링: 1인 사용자에게 진화적 다양성 유지는 학술적 의미 이상의 실용적 가치가 제한적
- 실제 적용 피드백까지 시간 소요 — 시뮬레이션 평가의 신뢰성 문제

### Option B: 증분 단일 변종 개선

세션 분석 → 문제점/개선 기회 식별 → 단일 개선안 생성 → 적용 → 다음 세션들에서 효과 관찰.

**장점:**
- 최소 비용: 개선당 LLM 호출 1-2회
- 직관적 피드백 루프: 변경 → 관찰 → 학습이 명확함
- 구현 단순: 세션 분석 + diff 생성 + git commit으로 완결
- 실제 사용 기반 검증 — 시뮬레이션 없이 실전 데이터로 판단

**단점:**
- 지역 최적 함정: 작은 조정만 반복하면 근본적 구조 변화를 놓침
- 새 에이전트/스킬 생성 같은 "창발적" 개선이 어려움
- 탐색 없이 exploitation만 하므로 혁신적 개선 불가능
- 단일 변종이 퇴보하면 감지까지 여러 세션이 필요할 수 있음

### Option C: 하이브리드 (증분 기본 + 상황적 진화)

**기본 모드는 증분 개선**이되, 특정 트리거가 발동하면 진화 루프로 전환한다.

진화 루프 트리거 조건:
1. **정체 감지**: 최근 K 세션 동안 동일 에이전트/스킬에서 반복적 피드백 패턴 발견
2. **구조적 변화 요청**: 새 에이전트/스킬 생성, 오케스트레이션 정책 재설계
3. **사용자 명시 요청**: `$evolve --full` 같은 수동 트리거
4. **주기적 탐색**: 일정 기간(예: 2주)마다 전체 시스템 재평가

하이브리드 작동 방식:
- 일상: 세션 Stop hook → 분석 → 증분 개선 제안 → git commit
- 진화 모드: parallel-codex 인프라로 N개 변종 생성 → 비교 평가 → 최적 선택

**장점:**
- 비용 효율: 대부분의 개선은 저비용 증분으로 처리
- 탐색-활용 균형: 정체 시에만 비용이 높은 진화 루프 가동
- 기존 인프라 활용: parallel-codex worktree가 진화 루프의 변종 생성에 그대로 사용 가능
- 점진적 구현 가능: 증분 모드 먼저 구현, 진화 모드는 이후 추가

**단점:**
- 트리거 조건 설계가 핵심 난제 — 잘못된 트리거는 불필요한 진화 루프 발동 또는 필요한 시점을 놓침
- 두 모드의 코드 경로 유지 필요 — 복잡도 증가
- "정체" 판단 기준이 모호 — 명확한 메트릭 정의 필요

## Decision

**Option C (하이브리드)** 채택.

증분 개선을 기본 모드로 사용하되, 정체 감지 시 parallel-codex + Codex를 활용한 진화 루프를 실행한다. 진화 모드의 월 상한은 두지 않는다 (Codex 토큰이 충분하므로 비용 제약 없음).

근거:
1. 1인 도구에서 비용 대비 효과가 가장 높음. 대부분의 개선은 "이 프롬프트에서 이 부분이 약하다"와 같은 명확한 신호 기반이므로 증분으로 충분함.
2. 진화 루프는 parallel-codex 인프라가 이미 존재하므로 추가 구현 비용이 낮음.
3. HyperAgents의 핵심 통찰(아카이브를 통한 다양성 유지, 메타인지적 자기수정)은 하이브리드에서도 보존 가능.
4. Future direction인 "메타-메타 루프"를 위해 진화 모드는 반드시 필요하되, 처음부터 상시 가동할 필요는 없음.
5. Codex 토큰이 충분한 현재 상황에서 진화 모드 월 상한을 두지 않아 정체 감지 시 즉시 대응 가능.

[ASSUMPTION][C] 증분 모드를 Phase 1으로 먼저 구현하고, 진화 모드는 Phase 2에서 추가한다. 두 모드의 전환은 자동 트리거 + 수동 트리거 모두 지원한다.

## Rejected Alternatives

- **Option A** 단독 적용은 기각. 1인 개발자의 개인 도구에서 상시 진화 루프는 비용 대비 가치가 낮다. HyperAgents 논문의 실험 환경(다수 에이전트가 경쟁하는 환경)과 현재 시스템(단일 사용자의 도구 최적화)은 근본적으로 다른 문제 공간이다.
- **Option B** 단독 적용은 기각. 새 에이전트/스킬 생성, 오케스트레이션 정책 재설계 같은 구조적 변화를 증분만으로는 달성할 수 없다. "프로젝트 간 학습 전이"라는 future direction에도 탐색 메커니즘이 필수적이다.

## Consequences

### Positive
- 일상적 비용은 낮게 유지하면서 구조적 개선 능력 확보
- parallel-codex 인프라 재사용으로 구현 속도 향상
- 점진적 구현이 가능하여 Phase 1 출시까지의 리드타임 최소화
- 사용 패턴 데이터 축적 후 진화 모드 트리거를 데이터 기반으로 튜닝 가능

### Skeptic 보완: 커밋 반영률 보조 지표
- Performance Scorer에 git blame 기반 "에이전트 출력 -> 커밋 반영률" 보조 지표를 추가한다.
- "피드백 감소"만으로 개선 여부를 판단하면 확증 편향에 빠질 수 있다. 에이전트가 생성한 코드가 실제로 커밋에 얼마나 반영되었는지를 정량적으로 측정하여, 피드백 감소와 교차 검증한다.
- 이 지표는 증분 모드와 진화 모드 모두에서 변종 평가의 보조 fitness 메트릭으로 사용한다.

### Negative
- 두 모드 간 전환 로직 설계·유지 비용
- 진화 모드 트리거 조건의 false positive/negative 리스크
- 증분 모드와 진화 모드의 결과물 병합 정책 필요 (증분으로 조금씩 바뀐 것 위에 진화가 덮어쓸 때)

### Future work
- 정체 감지 메트릭 정의 (ADR-0004의 세션 분석 결과에 의존)
- 진화 모드에서 생성할 변종 수(N)의 동적 조정 — 비용 예산 기반
- 메타-메타 루프: 하이브리드 전략 자체의 트리거 조건을 자기개선

## Assumptions

1. [ASSUMPTION][C] parallel-codex의 worktree 인프라가 변종 생성에 그대로 사용 가능하다 — worktree 내 에이전트 프로파일 수정 + 테스트 실행이 독립적으로 동작해야 한다.
2. [ASSUMPTION][C] "정체"는 최근 N 세션에서 동일 카테고리의 피드백이 K회 이상 반복되는 것으로 조작적 정의 가능하다 — 구체적 N, K 값은 운영 데이터로 튜닝.
3. [ASSUMPTION][C] 증분 개선의 효과 측정은 "다음 세션에서 동일 패턴의 피드백 감소"로 판단 가능하다.
4. [ASSUMPTION][C] 진화 모드 1회 실행 비용은 LLM 호출 기준 증분 모드 4-8배 수준이며, 월 1-2회 이하로 발동될 것이다.

## Revisit Triggers

- 증분 모드만으로 3개월 운영 후에도 구조적 개선이 전혀 발생하지 않는 경우 → Option A 재검토
- LLM 호출 비용이 현저히 감소하여 상시 진화 루프가 경제적으로 실행 가능해지는 경우 → Option A 재검토
- 증분 모드의 개선 제안 채택률이 80% 이상으로 높아 진화 모드가 불필요하다고 판단되는 경우 → Option B 재검토
- 에이전트/스킬 수가 30개 이상으로 증가하여 증분 모드로는 전체 시스템 일관성 유지가 어려운 경우 → Option A 비중 증가 검토
