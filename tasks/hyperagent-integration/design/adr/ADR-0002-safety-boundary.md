# ADR-0002: Safety Boundary — 자기수정의 안전 경계

- Status: accepted
- Date: 2026-04-13
- Related docs: ../PRD.md, ../ARCHITECTURE.md, ADR-0001-evolution-strategy.md

## Context

자기개선 시스템의 핵심 리스크는 **퇴보(regression)**다. 에이전트 프로파일, 스킬 프롬프트, 오케스트레이션 정책을 자동으로 수정하는 시스템이 잘못된 변경을 적용하면:

- 에이전트가 기존에 잘 수행하던 작업을 더 이상 수행하지 못함
- 스킬 프롬프트가 핵심 지침을 잃어버려 출력 품질 저하
- 오케스트레이션 정책 변경으로 에이전트 간 협업이 깨짐
- 연쇄 퇴보: A 에이전트 변경이 B 에이전트에 의존하는 워크플로우를 깨뜨림

1인 개발자 도구이므로 "별도의 QA 팀이 검증"하는 경로는 없다. 안전 메커니즘은 **자동화되거나 최소한의 사용자 개입**으로 작동해야 한다.

### 현재 시스템의 안전 메커니즘

- agent-registry는 git으로 관리됨 → `git log`, `git diff`, `git revert`로 모든 변경 추적·복원 가능
- symlink 기반 배포 → 소스 파일 수정이 즉시 반영되므로, 잘못된 변경도 즉시 노출됨
- 기존 hooks 인프라(post-commit, post-merge 등)가 변경 후 검증에 활용 가능

### 자기수정 대상별 리스크 수준

| 대상 | 변경 빈도 | 영향 범위 | 리스크 |
|------|----------|----------|--------|
| 에이전트 instructions.md | 높음 | 해당 에이전트 | 중간 |
| 스킬 프롬프트 | 높음 | 해당 스킬 | 중간 |
| workflow.toml 정책 | 낮음 | 전체 시스템 | 높음 |
| 새 에이전트/스킬 생성 | 낮음 | 신규 추가 | 낮음 (기존에 영향 없음) |
| agent-registry 구조 변경 | 매우 낮음 | 전체 시스템 | 매우 높음 |

## Options Considered

### Option A: Git 기반 변경 추적 전용 (Git-tracked changes only)

모든 자기수정은 반드시 git commit으로 기록된다. 퇴보 감지 시 `git revert`로 롤백한다.

작동 방식:
1. 자기개선 에이전트가 변경 생성
2. 변경을 git commit (커밋 메시지에 `[self-improve]` 태그)
3. 다음 세션들에서 효과 관찰
4. 퇴보 감지 시 해당 커밋을 `git revert`

**장점:**
- 구현 극도로 단순 — 기존 git 인프라만 사용
- 완전한 변경 이력 — 어떤 변경이 언제 왜 일어났는지 추적 가능
- 롤백이 원자적 — 특정 개선 커밋만 정확히 되돌릴 수 있음
- symlink 배포와 자연스럽게 통합 — revert 즉시 반영

**단점:**
- **사후적(reactive) 안전**: 퇴보가 발생한 후에야 감지하고 롤백함. 발생과 감지 사이에 여러 세션이 지나갈 수 있음
- 퇴보 감지 자체가 어려움 — "이 세션이 나빠진 건 에이전트 변경 때문인가, 작업 난이도 때문인가?"
- 연쇄 변경 시 롤백이 복잡해짐 — A 변경 위에 B 변경이 쌓인 경우
- 잘못된 변경이 적용되는 순간부터 롤백까지 사용자 경험 저하

### Option B: 섀도우 배포 (Shadow deployment)

새 변종을 기존 버전과 병렬로 실행하여, 새 변종이 기존보다 나을 때만 승격한다.

작동 방식:
1. 현재 버전(champion)과 새 변종(challenger) 공존
2. 동일 입력에 대해 둘 다 실행 (또는 시뮬레이션)
3. 평가 기준으로 비교
4. challenger가 champion을 이기면 승격, 아니면 폐기

**장점:**
- **사전적(proactive) 안전**: 퇴보가 프로덕션에 도달하기 전에 차단
- A/B 테스트와 동일한 원리로 통계적 신뢰도 확보 가능
- 가장 높은 안전성 — 기존 버전은 challenger가 입증될 때까지 절대 변경되지 않음

**단점:**
- **비용 2배**: 모든 작업을 두 번 실행하거나, 별도 시뮬레이션 환경 필요
- 1인 도구에서 "동일 입력"을 재현하기 어려움 — 실제 사용은 매번 다른 컨텍스트
- 시뮬레이션 평가의 타당성 문제 — 실제 사용과 시뮬레이션의 괴리
- 구현 복잡도 높음 — 두 버전 병렬 실행 인프라 필요
- 승격 판단까지 시간이 오래 걸림 — 충분한 비교 데이터 필요

### Option C: 단계적 적용 (Staged rollout)

변경의 리스크 수준에 따라 적용 범위를 점진적으로 확대한다.

작동 방식:
1. **Tier 1 — 즉시 적용**: 새 에이전트/스킬 생성 (기존에 영향 없음)
2. **Tier 2 — 커밋 + 자동 관찰**: 개별 에이전트/스킬 프롬프트 수정. git commit으로 기록하고, 이후 3세션 동안 해당 에이전트의 성능 추적. 퇴보 감지 시 자동 revert.
3. **Tier 3 — 사용자 승인 필요**: workflow.toml 정책 변경, agent-registry 구조 변경. diff를 사용자에게 보여주고 승인 후 적용.

**장점:**
- 리스크 수준에 비례하는 보호 — 과잉 보호도, 과소 보호도 아님
- Tier 1/2는 자동으로 작동하여 사용자 개입 최소화
- Tier 3의 명시적 승인으로 시스템 전체에 영향을 미치는 변경에 대한 인간의 판단 보존
- git 기반 롤백과 자연스럽게 결합
- "관찰 기간" 동안의 세션 데이터가 자기개선의 추가 학습 데이터로 활용 가능

**단점:**
- Tier 분류 기준의 설계가 필요 — 경계 케이스 존재 (예: 스킬 프롬프트의 핵심 동작 변경은 Tier 2인가 3인가?)
- Tier 2의 "3세션 관찰" 동안 퇴보된 상태로 운영됨 (Option A의 문제를 축소한 것이지 제거한 것은 아님)
- Tier 3의 승인 요청이 빈번하면 사용자 피로 유발

## Decision

**Option C (3-Tier 단계적 적용)** 채택.

근거:
1. 리스크 비례 보호가 1인 개발자 시나리오에 가장 적합. 모든 변경에 동일한 보호를 적용하면 불필요한 비용이거나 불충분한 보호.
2. Tier 1(즉시 적용)은 새 에이전트/스킬 생성을 마찰 없이 허용하여 자기개선의 "창발적" 측면을 살림.
3. Tier 2(자동 관찰)는 대부분의 개선이 해당하는 구간이며, 사용자 개입 없이 퇴보 감지-롤백이 가능.
4. Tier 3(사용자 승인)는 시스템 전체 정책 변경에만 적용되어, 인간의 판단이 가장 필요한 곳에만 개입.
5. Git 기반 추적(Option A의 핵심)을 기본 레이어로 포함하므로, Option A의 장점을 잃지 않음.

[ASSUMPTION][C] Tier 2의 관찰 기간은 3세션으로 시작하고, 운영 데이터에 따라 조정한다. 퇴보 판단 기준은 ADR-0004의 세션 분석 메트릭에 의존한다.

## Rejected Alternatives

- **Option A** 단독은 기각. 사후적 안전만으로는 퇴보가 감지되기까지의 사용자 경험 저하를 방지할 수 없다. 다만 Option A의 핵심(git 기반 변경 추적)은 Option C의 기본 레이어로 포함된다.
- **Option B**는 기각. 1인 도구에서 섀도우 배포의 비용(2배 실행)은 과도하며, "동일 입력 재현"이 현실적으로 불가능하다. 에이전트의 품질은 정적 벤치마크가 아니라 실제 사용 컨텍스트에서만 제대로 평가할 수 있다.

## Consequences

### Positive
- 자기개선이 대부분 자동으로 작동하되, 위험한 변경에서만 사용자 개입 요구
- git 이력에 모든 자기개선 흔적이 남아 투명성 확보
- 퇴보 자동 감지-복원으로 시스템 안정성 유지
- Tier 분류 자체가 "어떤 변경이 위험한가"에 대한 명시적 지식 축적

### Negative
- Tier 분류 규칙 관리 비용 — 새로운 자기수정 대상이 추가될 때마다 Tier 지정 필요
- Tier 2의 관찰 기간 동안 일시적 퇴보 허용 (최대 3세션)
- Tier 3 승인 요청 UX 설계 필요 — CLI 기반에서 비동기 승인을 어떻게 처리할지

### Future work
- Tier 분류의 자동화: 변경의 영향 범위를 정적 분석으로 자동 판단
- 관찰 기간의 적응형 조정: 에이전트별 사용 빈도에 따라 세션 수 차등 적용
- Tier 3 → Tier 2 자동 강등: 특정 유형의 정책 변경이 반복적으로 안전하다고 판명되면 승인 없이 자동 적용

## Assumptions

1. [ASSUMPTION][C] Tier 분류는 변경 대상(파일 경로)으로 결정된다: `agents/*/instructions.md`, `skills/*/` → Tier 2, `policy/workflow.toml` → Tier 3, 신규 파일 생성 → Tier 1.
2. [ASSUMPTION][C] Tier 2의 퇴보 감지는 세션 내 사용자 피드백 패턴(수정 요청, 재시도, 불만 표현)의 빈도 증가로 판단한다.
3. [ASSUMPTION][C] Tier 3 승인은 다음 세션 시작 시 사용자에게 pending 변경 사항을 요약하고 승인/거부를 묻는 방식으로 처리한다.
4. [ASSUMPTION][C] 자기개선 커밋은 `[self-improve]` 태그를 포함하여 수동 커밋과 구분 가능하다.

## Revisit Triggers

- Tier 2의 자동 관찰 기간(3세션) 내 퇴보 감지율이 너무 낮은 경우 → 관찰 기간 연장 또는 Option B 부분 도입 검토
- Tier 3 승인 요청이 월 5회 이상 발생하여 사용자 피로가 심한 경우 → Tier 3 대상 축소 또는 자동 승인 조건 도입
- 연쇄 퇴보(A 변경 → B 의존 워크플로우 깨짐)가 발생하는 경우 → 의존성 그래프 기반 영향 분석 추가 검토
- 자기개선이 활발해져서 Tier 2 커밋이 일 3건 이상 발생하는 경우 → 배치 커밋으로 전환 검토
