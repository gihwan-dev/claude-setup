# ADR-0004: Session as Event Log — 세션 데이터를 자기개선의 핵심 소스로 활용

- Status: accepted
- Date: 2026-04-13
- Related docs: ../PRD.md, ../ARCHITECTURE.md, ADR-0001-evolution-strategy.md, ADR-0002-safety-boundary.md, ADR-0003-variant-management.md

## Context

Anthropic의 Managed Agents 아키텍처에서 **세션(Session)은 append-only 이벤트 로그**다. 모든 메시지, 도구 호출, 결과가 불변(immutable)하게 기록된다. 이 설계는 디버깅, 감사(audit), 재현(replay)을 가능하게 한다.

현재 Claude Code는 이미 풍부한 세션 데이터를 저장하고 있다 (spike-3에서 확인):

| 데이터 소스 | 위치 | 내용 |
|------------|------|------|
| 세션 JSONL | `~/.claude/projects/<project>/<uuid>.jsonl` | 대화 전체: 메시지, 도구 호출, 결과, 타임스탬프, git branch |
| 글로벌 히스토리 | `~/.claude/history.jsonl` (4.1MB) | 프로젝트별 명령어 히스토리, sessionId |
| 통계 캐시 | `~/.claude/stats-cache.json` | 일별 활동, 모델별 토큰, 총 2175 세션/288707 메시지 |
| 활성 세션 | `~/.claude/sessions/` | 현재 실행 중인 세션 메타데이터 |

이 데이터에서 자기개선 신호를 추출할 수 있다:

- **부정 신호**: 사용자의 수정 요청, 재시도, "아니", "다시", "잘못됐다" 등의 표현
- **긍정 신호**: 에이전트 출력을 수정 없이 수용, "좋다", "감사" 등의 표현
- **구조 신호**: 특정 에이전트/스킬의 호출 빈도, 성공률, 토큰 사용량
- **패턴 신호**: 반복적으로 같은 유형의 수정 요청이 발생하는 경우

핵심 질문: **이 데이터를 언제, 어떻게 분석하여 자기개선에 투입할 것인가?**

### 분석 비용 고려

세션 JSONL 하나의 크기는 수십 KB에서 수 MB까지 다양하다. 2175개 세션의 전체 분석은 비현실적이다. 분석 자체가 LLM 호출을 필요로 하므로 비용이 발생한다.

## Options Considered

### Option A: 배치 분석 (Periodic batch analysis)

주기적으로(예: 매일 자정, 또는 매주 일요일) 최근 세션들을 일괄 분석한다.

작동 방식:
1. cron 또는 launchd로 스케줄된 분석 스크립트 실행
2. 마지막 분석 이후의 세션 JSONL 수집
3. LLM에 세션 요약 + 피드백 패턴 분석 요청
4. 개선 제안을 생성하여 `pending-improvements.jsonl`에 저장
5. 다음 세션 시작 시 pending 개선 적용 (ADR-0002의 Tier 분류에 따라)

**장점:**
- 여러 세션을 한꺼번에 분석하여 패턴 감지 정확도 향상 — 단일 세션의 노이즈 제거
- LLM 호출을 배치로 묶어 비용 효율화 (프롬프트 캐싱 활용 가능)
- 세션 실행 중 추가 지연 없음 — 분석은 오프라인에서 수행
- 장기 추세 분석에 적합 — "최근 2주간 architecture-reviewer의 피드백률 증가 추세"

**단점:**
- **지연**: 개선이 발생하려면 다음 배치 분석까지 대기해야 함. 일 1회 분석이면 최대 24시간 지연
- 배치 분석 시점에 컴퓨터가 꺼져 있으면 스킵됨 — 개인 Mac에서 cron 안정성 문제
- 대량 세션 분석의 LLM 비용이 한 번에 집중 — 비용 예측 어려움
- 세션 컨텍스트가 시간이 지나면 희석됨 — "왜 이 피드백을 줬는지" 맥락 손실

### Option B: 훅 기반 실시간 수집 (Hook-driven real-time collection)

Claude Code의 Stop hook에서 세션 종료 시 즉시 분석을 트리거한다.

작동 방식:
1. `~/.claude/settings.json`에 Stop hook 등록
2. 세션 종료 시 hook이 `analyze-session.sh` 실행
3. 방금 끝난 세션의 JSONL을 분석하여 피드백 신호 추출
4. 즉시 개선 제안 생성 또는 신호를 `signals.jsonl`에 누적
5. 임계치 도달 시 개선 제안 생성

**장점:**
- **즉시성**: 세션 종료 직후 분석하므로 컨텍스트 손실 최소
- 세션별 분석이므로 개별 세션의 미세한 신호도 포착 가능
- cron 의존 없음 — 사용 패턴에 자연스럽게 동기화
- 점진적 신호 누적: 매 세션마다 가벼운 분석, 임계치 도달 시만 본격 개선

**단점:**
- **세션 종료 시 지연**: 분석 스크립트 실행 시간이 사용자 경험에 영향 (hook이 동기 실행되는 경우)
- 단일 세션 분석의 노이즈: 특정 세션이 비정상적으로 어려운 작업이었을 수 있음
- 매 세션마다 LLM 호출 비용 발생 — 세션당 분석 비용 × 일일 세션 수
- Stop hook의 안정성: hook 실패 시 세션 데이터 유실 가능
- 세션이 짧으면(질문 1-2개) 분석 가치 대비 비용 비효율

### Option C: 온디맨드 분석 (On-demand analysis)

사용자 또는 메타에이전트가 명시적으로 분석을 요청할 때만 실행한다.

작동 방식:
1. `$evolve analyze` 명령으로 사용자가 수동 트리거
2. 또는 메타에이전트가 ADR-0001의 하이브리드 전략에 따라 자동 트리거
3. 분석 범위를 지정 가능: 최근 N 세션, 특정 에이전트/스킬, 특정 기간
4. 분석 결과를 즉시 보여주고 개선 제안 제시

**장점:**
- 완전한 사용자 통제 — 원할 때만 분석, 비용 예측 가능
- 분석 범위를 유연하게 지정 가능
- 불필요한 분석 비용 제거 — 문제를 느낄 때만 분석
- 구현이 가장 단순 — 스킬 하나로 구현 가능

**단점:**
- **수동 의존**: 사용자가 분석을 잊으면 자기개선이 멈춤 — "자동"이 아님
- 자기개선의 핵심 가치(알아서 좋아지는 것)와 모순
- 사용자가 문제를 인식하지 못하는 미세한 퇴보를 잡지 못함
- 반복적 수동 트리거는 결국 자동화를 원하게 됨

### Option D: 하이브리드 — 경량 훅 + 주기적 심층 분석

Option B와 A를 결합한 2단계(two-tier) 분석.

작동 방식:

**1단계 (경량, 매 세션 종료):**
- Stop hook에서 LLM 없이 **규칙 기반** 경량 분석 실행
- 세션 JSONL에서 정량적 신호만 추출: 도구 호출 성공/실패율, 사용자 수정 횟수, 토큰 수, 세션 시간
- 결과를 `signals.jsonl`에 append
- 실행 시간: 1초 미만 (jq/python 스크립트, LLM 호출 없음)

**2단계 (심층, 주기적 또는 트리거 기반):**
- signals.jsonl에 누적된 정량 데이터를 기반으로 "분석 필요" 판단
- 트리거 조건: 특정 에이전트의 피드백률이 기준 초과, 또는 최근 N 세션 분석 미실행
- LLM 기반 심층 분석: 세션 JSONL의 실제 대화 내용에서 정성적 패턴 추출
- 개선 제안 생성 → ADR-0002의 Tier 분류에 따라 적용

**장점:**
- 1단계는 비용 제로, 지연 제로 — 매 세션 자동 수집
- 2단계는 필요할 때만 실행되어 비용 효율적
- 정량 + 정성 분석의 결합으로 가장 풍부한 신호
- Stop hook 실패 시에도 2단계가 독립적으로 동작 가능
- signals.jsonl이 그 자체로 시스템 건강 대시보드 역할

**단점:**
- 2개 시스템(경량 수집기 + 심층 분석기) 구현·유지 필요
- 1단계의 규칙 기반 신호 추출 규칙 설계 필요
- 2단계 트리거 조건 설계 (ADR-0001의 정체 감지와 동일한 난제)
- signals.jsonl 스키마 설계 및 관리

### Option E: 일일 배치 cron (Daily batch cron)

hooks 방식을 제거하고, 매일 지정 시각(예: 22:00)에 당일 세션 JSONL을 직접 분석하는 단순한 배치 방식.

작동 방식:
1. launchd(macOS) 또는 cron으로 매일 22:00에 분석 스크립트 실행
2. 당일 생성/수정된 세션 JSONL 파일을 수집
3. LLM 기반 분석: 세션별 피드백 신호 추출 + 에이전트/스킬별 성능 집계
4. 개선 제안 생성 -> ADR-0002의 Tier 분류에 따라 적용 또는 pending
5. `$evolve` CLI로 수동 실행도 지원 — 원하는 시점에 즉시 분석 가능

signals.jsonl 중간 저장소는 사용하지 않는다. 세션 JSONL 자체가 원본 데이터이므로 중간 저장소 없이 직접 분석한다.

**장점:**
- **구현 단순**: hooks 인프라, signals.jsonl 스키마, 2단계 트리거 조건 등 복잡한 설계가 불필요
- 세션 JSONL이 유일한 데이터 소스 — 중간 저장소 동기화 문제 없음
- 여러 세션을 한꺼번에 분석하여 단일 세션 노이즈 제거
- `$evolve` CLI로 수동 실행 가능하여 사용자 통제 유지
- 세션 종료 시 추가 지연 없음 — 분석은 별도 시점에 오프라인 실행
- cron이 실패해도 다음 날 또는 수동 실행으로 보완 가능

**단점:**
- 최대 24시간 지연 — 세션 종료 후 개선까지 시간 소요
- Mac 슬립 시 launchd 스케줄 스킵 가능 (wake-on-schedule 설정 또는 수동 보완 필요)
- 매일 LLM 분석 비용 발생 (단, 일일 1회로 예측 가능)
- 세션 직후 컨텍스트가 가장 생생한 시점을 활용하지 못함

## Decision

**Option E (일일 배치 cron)** 채택.

hooks 방식을 삭제하고, 매일 지정 시각(예: 22:00)에 당일 세션 JSONL을 직접 분석하는 일일 배치 방식을 채택한다. signals.jsonl 중간 저장소는 제거한다. `$evolve` CLI로 수동 실행도 지원한다.

근거:
1. hooks 인프라(Stop hook -> signals.jsonl -> 2단계 트리거)는 설계 복잡도 대비 실질적 이점이 제한적이다. 1인 개발자 환경에서 세션 직후 즉시 분석의 가치는 크지 않다.
2. signals.jsonl 중간 저장소를 제거하면 스키마 설계, 동기화, 크기 관리 등 부수 복잡도가 사라진다. 세션 JSONL 자체가 원본 데이터로 충분하다.
3. 일일 배치는 여러 세션을 한꺼번에 분석하므로 단일 세션의 노이즈를 제거하고 일일 추세를 파악할 수 있다.
4. `$evolve` CLI 수동 실행으로 사용자가 원하는 시점에 즉시 분석을 트리거할 수 있어, cron 의존성을 보완한다.
5. ADR-0001(하이브리드 전략)과 정렬: 일일 배치 분석 결과가 "정체 감지"의 데이터 소스 역할.
6. ADR-0003(변종 관리)과 정렬: 분석 결과가 archive.jsonl의 fitness 메트릭에 직접 반영.

## Rejected Alternatives

- **Option A** (배치 전용)는 기각. Option E가 배치 방식을 채택하되 `$evolve` CLI 수동 실행을 병행하여 cron 의존성을 보완한다. Option A의 핵심 아이디어는 Option E에 흡수.
- **Option B** (훅 기반 실시간 전용)는 기각. 매 세션마다 LLM 호출은 비용 과다이며, 단일 세션의 노이즈 문제를 해결하지 못한다. hooks 인프라의 설계 복잡도 대비 실질적 이점이 제한적이다.
- **Option C** (온디맨드 전용)는 기각. "자동으로 알아서 좋아지는" 시스템이라는 핵심 목표와 모순된다. 다만 `$evolve` 수동 트리거는 Option E에서도 지원.
- **Option D** (하이브리드 -- 경량 훅 + 주기적 심층 분석)는 기각. 2단계 시스템(경량 수집기 + 심층 분석기)과 signals.jsonl 중간 저장소의 설계·유지 복잡도가 1인 개발자 환경에서 과도하다. 세션 JSONL 직접 분석으로 동일한 목표를 더 단순하게 달성 가능.

## Consequences

### Positive
- 일일 배치로 당일 전체 세션을 한꺼번에 분석 — 단일 세션 노이즈 제거, 일일 추세 파악 가능
- 중간 저장소(signals.jsonl) 제거로 아키텍처 단순화 — 세션 JSONL이 유일한 데이터 소스
- `$evolve` CLI 수동 실행으로 사용자 통제와 자동화를 균형있게 유지
- 세션-변종 간 인과 연결 가능 — "이 변경 이후 피드백률이 어떻게 변했는가?"
- LLM 분석 비용이 일일 1회로 예측 가능

### Skeptic 보완: 작업 난이도 통제
- Session Analyzer가 세션별 작업 복잡도를 태깅한다 (예: 단순 질의, 버그 수정, 신규 기능 개발, 아키텍처 설계 등).
- 동일 에이전트라도 쉬운 작업과 어려운 작업에서 피드백 패턴이 다르다. 난이도를 통제하지 않으면 "어려운 작업이 많았던 날"을 "에이전트 퇴보"로 오판하는 확증 편향이 발생한다.
- 난이도 태그를 기준으로 동일 난이도 그룹 내에서만 성능 추이를 비교하여 정확한 퇴보/개선 판단을 지원한다.

### Negative
- 최대 24시간 분석 지연 — 즉시 피드백 반영이 불가능
- LLM 분석 프롬프트 설계에 분석 품질이 크게 의존
- Mac 슬립 시 launchd 스케줄 스킵 가능성 — 수동 보완 필요

### Future work
- 분석 결과 기반 대시보드 (CLI 또는 웹) — 에이전트/스킬별 건강 추이 시각화
- 분석 프롬프트 자동 개선 — 메타-메타 루프의 일부
- 프로젝트 간 세션 분석 통합 — 교차 프로젝트 패턴 발견
- 작업 난이도 자동 태깅 정확도 향상 — 초기 LLM 추론 기반에서 데이터 축적 후 규칙 기반으로 전환

## Assumptions

1. [ASSUMPTION][E] 일일 배치 분석에서 추출할 초기 신호 목록:
   - 세션 시간(분)
   - 총 메시지 수 / 사용자 메시지 수
   - 도구 호출 총 횟수 / 실패 횟수
   - 에이전트/스킬 식별자 (어떤 에이전트/스킬이 사용되었는지)
   - 사용자 수정 패턴 횟수 (연속된 짧은 사용자 메시지 = 수정 지시일 확률 높음)
   - 최종 결과 수용 여부 (세션 마지막 메시지가 사용자의 긍정적 반응인지)
   - 작업 난이도 태그 (단순 질의 / 버그 수정 / 신규 기능 / 아키텍처 설계 등)
2. [ASSUMPTION][E] 일일 배치 실행 시각은 22:00으로 설정하되, launchd plist로 관리한다. Mac 슬립 복귀 시 missed schedule은 즉시 실행되도록 설정.
3. [ASSUMPTION][E] 세션 JSONL에서 에이전트/스킬 식별은 도구 호출의 패턴(TeamCreate, Skill 호출 등)에서 추론 가능하다.
4. [ASSUMPTION][E] `$evolve` CLI는 수동 실행 시 분석 범위를 지정 가능: 최근 N일, 특정 에이전트/스킬, 특정 기간.
5. [ASSUMPTION][E] 일일 분석의 LLM 비용은 당일 세션 수에 비례하며, 일일 10세션 기준 1회 분석당 Codex 호출 1-2회 수준으로 제한.

## Revisit Triggers

- 일일 배치 분석의 개선 제안 품질이 낮아 실제 적용률이 30% 미만인 경우 -> 분석 프롬프트 재설계 또는 실시간 hooks 방식(Option D) 재검토
- launchd 스케줄 스킵이 빈번하여 분석 누락이 주 2회 이상 발생하는 경우 -> `$evolve` 자동 실행 보완 또는 hooks 방식 부분 도입
- LLM 분석 비용이 예상보다 높아 일일 실행이 부담되는 경우 -> 주간 배치로 전환 검토
- 세션 JSONL 포맷이 Claude Code 업데이트로 변경되는 경우 -> 분석 파서 업데이트 필요
- 프로젝트 수가 증가하여 교차 프로젝트 분석이 필요해지는 경우 -> 통합 분석 아키텍처 설계
- 작업 난이도 태깅의 정확도가 낮아 성능 비교가 왜곡되는 경우 -> 태깅 로직 개선 또는 수동 태깅 보완
