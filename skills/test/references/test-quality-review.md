# Test Quality Review Guide

## Test Value Evaluation Framework

Evaluate tests using Khorikov's 4 Pillars.

| Pillar | Question | Note |
|------|------|------|
| **Regression protection** | How likely is this test to catch a real bug? | It should execute non-trivial paths in production code |
| **Resistance to refactoring** | Does it avoid breaking unnecessarily when implementation changes? | **Non-negotiable** — if this pillar is weak, the test is bad no matter how strong the others are |
| **Fast feedback** | Does the execution speed preserve the development loop? | Slow tests break the feedback loop |
| **Maintainability** | Is it cheap to read, change, and understand? | Excessive setup or unclear intent lowers maintainability |

It is impossible to maximize all four pillars at once. In practice, keep **resistance to refactoring fixed** and trade off across the other three.

- **Unit tests**: regression protection ↓ + fast feedback ↑ + maintainability ↑
- **Integration / browser tests**: regression protection ↑ + fast feedback ↓ + maintainability ↓

## Typical Problems in AI-Generated Tests

### Tautological assertion

An assertion that duplicates the production logic in the expected value. When the implementation changes, the test must change with it, so the test offers zero regression protection.

**How to detect it:**
- The expected value is generated by calling a production function (for example `expect(result).toBe(calculateTotal(items))`)
- The expected value recreates the same branches as production code
- Deleting the test would not reduce the ability to detect a production-breaking change

### Hallucinated constraint

A test that asserts behavior or constraints that do not exist in the code. The AI guessed that "there is probably a rule like this."

**How to detect it:**
- The asserted behavior is not implemented or guaranteed in production code
- The test passes only because "that value happened to come out"
- The rule is not documented in code comments or specs

### Excessive mocking

A test where the number of mocked dependencies exceeds the number of meaningful assertions. The test ends up validating "was the mock configured correctly?" instead of real behavior.

**How to detect it:**
- Number of `jest.mock()` / `vi.mock()` / `sinon.stub()` calls > number of `expect()` calls
- Mock setup accounts for more than 50% of the test code
- Mock return values do not structurally match real production return values

### Implementation mirroring

A pattern where the production branch logic is rebuilt inside the test. If production code is wrong, the test becomes wrong in the same way and fails to catch the bug.

**How to detect it:**
- The test contains the same `if` / `switch` branches as production code
- A test helper is effectively a miniature version of the production function
- The "expected" value is derived by calculation rather than being a deliberate hard-coded expectation

## keep / rewrite / merge / delete Decision Flow

Evaluation order: **delete → merge → rewrite → keep**. Start with the most aggressive decision so low-value tests are filtered out first.

### Step 1: delete conditions

Return **delete** if any of the following apply:

- **Tautological**: The expected value duplicates production logic → regression protection value is 0
- **Liar test**: It always passes but validates nothing (empty assertions, meaningless truthy checks)
- **Framework-testing**: It retests framework / library internals (see anti-pattern #1 in `testing-principles.md`)
- **Dead-code test**: It tests deleted features or unreachable code

### Step 2: merge opportunities

For tests that are not delete candidates, return **merge** when:

- **Scenario duplication**: Two or more tests validate the same input-action-result path
- **Behavior fragmentation**: Assertions for one user behavior are scattered across multiple tests
- **Incremental duplication**: Test A is a strict subset of Test B

For merge candidates, specify which tests should be merged and in what direction.

### Step 3: rewrite conditions

For tests that are neither delete nor merge candidates, return **rewrite** when:

- **Excessive mocking**: Mocking replaces the core logic itself (see anti-pattern #5 in `testing-principles.md`)
- **Implementation coupling**: The test inspects internal state / methods directly (see anti-pattern #6 and section 3 in `testing-principles.md`)
- **Weak assertions**: Low-specificity assertions such as `toBeTruthy()` and `toBeDefined()` (see section 8 in `testing-principles.md`)
- **Snapshot overuse**: Snapshots include large unrelated surfaces (see anti-pattern #2 in `testing-principles.md`)
- **Hallucinated constraint**: The test asserts constraints that do not exist in the code

For rewrite candidates, explain the root problem and the direction of improvement.

### Step 4: keep

If none of the categories above apply, return **keep**. These are tests that preserve resistance to refactoring and provide meaningful regression protection under the 4 Pillars.
