# 테스트 품질 리뷰 가이드

## 테스트 가치 평가 프레임워크

Khorikov의 4 Pillars를 기반으로 테스트를 평가한다.

| 기둥 | 질문 | 비고 |
|------|------|------|
| **회귀 보호** | 이 테스트가 실제 버그를 잡을 확률은? | 프로덕션 코드의 비자명(non-trivial) 경로를 실행해야 한다 |
| **리팩토링 저항성** | 구현 변경 시 불필요하게 깨지지 않는가? | **타협 불가** — 이 기둥이 낮으면 다른 기둥이 아무리 높아도 나쁜 테스트 |
| **빠른 피드백** | 실행 속도가 개발 루프를 방해하지 않는가? | 느린 테스트는 피드백 루프를 끊는다 |
| **유지보수성** | 읽기/수정/이해 비용이 낮은가? | 셋업이 과도하거나 의도가 불명확하면 유지보수성이 낮다 |

4 기둥을 모두 만족하는 것은 불가능하다. 실제로는 **리팩토링 저항성을 고정**하고 나머지 셋 사이에서 트레이드오프한다.

- **유닛 테스트**: 회귀 보호 ↓ + 빠른 피드백 ↑ + 유지보수성 ↑
- **통합/브라우저 테스트**: 회귀 보호 ↑ + 빠른 피드백 ↓ + 유지보수성 ↓

## AI 생성 테스트의 전형적 문제

### Tautological assertion

프로덕션 코드 로직을 테스트 기대값에 그대로 복제한 단언. 구현이 바뀌면 테스트도 같이 바뀌어야 하므로 회귀 보호 가치가 0이다.

**탐지법:**
- 기대값이 프로덕션 함수를 호출해 생성됨 (예: `expect(result).toBe(calculateTotal(items))`)
- 기대값이 프로덕션 코드의 조건 분기를 그대로 재현
- 테스트를 삭제해도 프로덕션 코드를 깨뜨릴 수 있는 변경이 검출되지 않음

### 환각된 규칙 (hallucinated constraint)

코드에 존재하지 않는 행동이나 제약을 단언하는 테스트. AI가 "아마 이런 규칙이 있을 것"이라 추측해 만든다.

**탐지법:**
- 단언하는 행동이 프로덕션 코드에서 구현/보장되지 않음
- 테스트가 통과하는 이유가 "우연히 그 값이 나와서"
- 코드 주석이나 스펙에 해당 규칙이 명시되지 않음

### 과도한 모킹

모킹 의존성 수가 실제 단언 수를 초과하는 테스트. 테스트가 검증하는 것은 "목이 올바르게 설정되었는가"가 되어 실제 행동 검증 가치가 급감한다.

**탐지법:**
- `jest.mock()` / `vi.mock()` / `sinon.stub()` 호출 수 > `expect()` 호출 수
- 모킹 셋업이 테스트 코드의 50% 이상
- mock 반환값이 실제 프로덕션 반환값과 구조적으로 다름

### 구현 복제 (implementation mirroring)

프로덕션의 분기 로직을 테스트에 그대로 재현하는 패턴. 프로덕션 코드가 틀려도 테스트가 같이 틀리므로 버그를 잡지 못한다.

**탐지법:**
- 테스트 내부에 프로덕션과 동일한 if/switch 분기가 존재
- 테스트 헬퍼가 프로덕션 함수의 축소판
- "expected" 값이 하드코딩이 아닌 계산식으로 도출

## keep/rewrite/merge/delete 판정 플로차트

판정 순서: **delete → merge → rewrite → keep**. 가장 과감한 판정부터 평가해 불필요한 테스트를 먼저 걸러낸다.

### 1단계: delete 조건

다음 중 하나라도 해당하면 **delete** 판정:

- **Tautological**: 프로덕션 로직을 복제한 기대값 → 회귀 보호 가치 0
- **Liar test**: 항상 통과하지만 검증하는 행동이 없음 (빈 단언, 무의미한 truthy 체크)
- **Framework-testing**: 프레임워크/라이브러리 내부 동작을 재검증 (→ `testing-principles.md` 안티패턴 #1 참조)
- **Dead-code test**: 삭제된 기능이나 도달 불가능한 코드를 테스트

### 2단계: merge 가능성

delete가 아닌 테스트 중 다음에 해당하면 **merge** 판정:

- **시나리오 중복**: 두 개 이상의 테스트가 동일한 입력-행동-결과 경로를 검증
- **동일 행동 분산**: 하나의 사용자 행동에 대한 검증이 여러 테스트에 흩어져 있음
- **단계적 중복**: 테스트 A가 테스트 B의 strict subset

merge 대상은 통합할 테스트 그룹과 통합 방향을 제시한다.

### 3단계: rewrite 필요성

merge가 아닌 테스트 중 다음에 해당하면 **rewrite** 판정:

- **과도한 모킹**: 모킹이 핵심 로직까지 대체 (→ `testing-principles.md` 안티패턴 #5 참조)
- **구현 결합**: 내부 상태/메서드를 직접 검사 (→ `testing-principles.md` 안티패턴 #6, 섹션 3 참조)
- **부실 단언**: `toBeTruthy()`, `toBeDefined()` 등 구체성 부족 (→ `testing-principles.md` 섹션 8 참조)
- **스냅샷 남용**: 변경과 무관한 부분까지 포함된 스냅샷 (→ `testing-principles.md` 안티패턴 #2 참조)
- **환각된 규칙**: 코드에 없는 제약을 단언

rewrite 대상은 문제 원인과 개선 방향을 제시한다.

### 4단계: keep

위 어디에도 해당하지 않으면 **keep** 판정. 4 Pillars 중 리팩토링 저항성이 확보되고, 의미 있는 회귀 보호를 제공하는 테스트.
