# 워크플로우: 프로젝트 마일스톤 생성

**목표**: 사용자가 전달한 기준 문서(예: 스펙, PRD, 회의록, 이슈, 요구사항 문서)를 분석해, 개발 진행 상황을 추적할 수 있는 **논리적이고 실행 가능한 작업 목록(Milestone)**을 `milestone.md`로 생성하십시오. 또한 **독립적으로 수행 가능한 Phase는 병렬 실행 표기**를 포함하십시오.

## 1단계: 기준 문서 확정

입력 문서 선택 우선순위:

1. 사용자 메시지에서 직접 지정한 문서/경로
2. `milestone.md`가 이미 있다면 상단의 "기반 문서" 목록
3. 프로젝트 문서 후보 (`SPEC.md`, `requirements.md`, `prd.md`, `README.md`, `docs/**`, 이슈/회의 노트 등)

문서 탐색 규칙:
- 가능한 한 **사용자가 지정한 문서를 우선**합니다.
- 문서가 여러 개인 경우 중복/충돌 요구사항을 정리해 기준 목록을 확정합니다.
- 기준 문서를 찾지 못하면 최소 질문 1개로 문서 경로를 확인합니다.

분석 항목:
1. **핵심 목표 식별**: 달성해야 할 사용자 가치와 주요 기능을 파악합니다.
2. **의존성 분석**: 선행이 필요한 흐름(예: 데이터 계약 -> API -> UI)을 파악합니다.
3. **작업 단위 정의**: 파일 단위가 아닌, **"의미 있는 결과가 검증 가능한가?"** 기준으로 정의합니다.
4. **병렬 가능성 식별**: 각 Phase가 다른 Phase 산출물 없이 독립 수행 가능한지 판정합니다.
5. **비기능 요구 반영**: 성능/접근성/보안/테스트 요구를 누락하지 않도록 포함합니다.

## 2단계: 컨텍스트 윈도우 친화 분해
최신 AI 모델의 컨텍스트 윈도우를 고려하여, 각 작업을 **한 번의 구현 세션에서 처리 가능한 크기**로 분해하십시오.

분해 규칙:
1. 한 항목은 단일 목표만 다룹니다.
2. 한 항목의 완료 조건(검증)은 최대 3개로 제한합니다.
3. 범위가 큰 항목은 수직 슬라이스(기능별)로 다시 분해합니다.
4. 의존성이 있으면 선행 항목을 분리하고, 후행 항목에 연결 관계를 명시합니다.
5. 항목 단위가 커서 "구현 + 검증 + 리뷰"를 한 번에 끝내기 어렵다면 반드시 추가 분할합니다.

권장 단위:
- 조사/불확실성 해소가 필요하면 `스파이크(조사)`를 별도 항목으로 둡니다.
- 구현 항목은 "동작 가능한 결과 + 검증 가능성"이 한 묶음이 되도록 작성합니다.

## 3단계: 마일스톤 구성 전략
아키텍처 계층보다는 **기능 완성도와 의존성** 중심으로 Phase를 구성하십시오.

병렬 그룹 규칙:
- 서로 독립 실행 가능한 Phase는 같은 병렬 그룹(`PG-1`, `PG-2`)으로 묶습니다.
- 병렬 판정 기준:
  - 선행 결과(타입/함수/API) 없이 시작 가능한가?
  - 동일 파일/모듈 동시 수정 충돌 위험이 낮은가?
  - 데이터/이벤트 흐름이 강결합되지 않았는가?

## 4단계: milestone.md 작성
다음 형식을 사용하여 `milestone.md`를 생성하십시오.

* 각 항목은 명확한 **'완료 조건'**을 포함해야 합니다.
* 구체적인 파일 경로나 아키텍처 용어보다 **개발 결과**에 집중하십시오.
* 각 Phase/항목이 기준 문서의 요구사항에 자연스럽게 대응되도록 구성하십시오.
* 각 Phase 제목에는 반드시 실행 라벨을 붙이십시오:
  * 병렬 가능: `[PARALLEL:PG-<번호>]`
  * 순차 전용: `[SEQUENTIAL]`
* 같은 `PG` 번호를 가진 Phase들은 서로 병렬 실행 가능한 묶음을 의미합니다.

### 출력 형식 예시

```markdown
# 프로젝트 개발 마일스톤

> 기반 문서: docs/product-requirements.md, docs/tech-notes.md
> 실행 라벨: `[SEQUENTIAL]`(순차), `[PARALLEL:PG-n]`(같은 그룹끼리 병렬)
> 분해 기준: 컨텍스트 윈도우 친화 단위(한 번에 구현/검증 가능한 크기)

## Phase 1. [SEQUENTIAL] 프로젝트 기반 및 공통 모듈
- [ ] **핵심 데이터 모델링 및 API 클라이언트 구성**
    - 목표: User, Product 등 전역에서 사용되는 핵심 타입 정의 및 API 통신 환경 구축
    - 검증: API 호출 테스트 성공 및 타입 추론 동작 확인

## Phase 2. [PARALLEL:PG-1] 핵심 기능 개발 - 사용자 인증
- [ ] **사용자 인증(로그인) 기능 구현**
    - 목표: 로그인 폼, 인증 API 연동, 세션 상태 반영까지 완료
    - 검증: 정상 로그인/실패 케이스 처리 확인

## Phase 3. [PARALLEL:PG-1] 핵심 기능 개발 - 상품 검색
- [ ] **상품 검색 기능 구현**
    - 목표: 검색 입력, 결과 조회 API 연동, 목록 UI 렌더링 완료
    - 검증: 키워드 변경 시 결과가 정상 갱신되는지 확인

## Phase 4. [SEQUENTIAL] 페이지 조립 및 배포 준비
- [ ] **상품 결제 프로세스 구현**
    - 목표: 장바구니 담기부터 결제 승인까지의 로직 및 UI 컴포넌트 개발
    - 포함: 결제 모달 UI, PG사 연동 로직, 주문 데이터 업데이트
    - 검증: 결제 성공 시 주문 내역 생성 확인

- [ ] **마이페이지 대시보드 완성**
    - 목표: 프로필 수정, 주문 내역 조회 등 개별 기능을 한 페이지에서 통합
    - 작업: 기존 개발된 위젯들을 조합하여 레이아웃 구성

```

## 실행 지시

위 논리를 바탕으로, 개발자가 **작업 흐름과 병렬 실행 가능 구간을 한눈에 파악할 수 있는** 실용적인 `milestone.md`를 생성하십시오. 특히 각 항목은 최신 AI 모델이 한 번에 구현/검증 가능한 크기로 세분화하십시오.
