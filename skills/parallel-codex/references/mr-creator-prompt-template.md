# MR Creator Prompt Template

특정 그룹의 task MR을 생성하는 Codex 프롬프트 구조.
그룹별로 점진적으로 호출된다 (일괄 생성이 아님).
Claude가 변수를 치환한 뒤 Codex에 전달한다.

## Template

```xml
<task>
## 목적
${GROUP_NAME}의 완료된 작업 브랜치들에 대해 개별 draft MR을 생성한다.
모든 MR은 main을 타겟으로 한다.

## 플랫폼
${PLATFORM}

## MR 생성 대상
${MR_ENTRIES}

## MR 생성 규칙
- 모든 MR은 **draft**로 생성
- MR 제목: `<type>(<scope>): <설명>` (한국어, 70자 이내)
- 각 MR은 해당 task의 변경사항만 설명
- MR body 포맷 (아래 구조대로 작성):

**MR Body:**
> ## Task
> - **ID**: (task-id)
> - **Group**: (group-name)
> - **Description**: (task-description)
>
> ## Merge Order
> 이 MR은 ${MERGE_ORDER_NOTE}
>
> 머지 순서 상세는 PIPELINE.md 참조.
>
> ---
> *Generated by parallel-codex pipeline*

## 완료 기준
- 대상 task 브랜치 전부에 대해 draft MR 생성
- 각 MR의 URL 수집
- PIPELINE.md의 MR Plan 상태 업데이트
</task>

<structured_output_contract>
완료 후 반드시:
1. 생성된 MR 목록 (번호, 제목, URL)
2. PIPELINE.md의 MR Plan 테이블 업데이트 (Status를 `created`로, URL 추가)
3. PIPELINE.md의 Groups 테이블에서 해당 그룹 Status를 `mr_created`로 업데이트
</structured_output_contract>

<action_safety>
- MR 생성과 PIPELINE.md 업데이트만 수행
- 코드 수정 금지
- 브랜치 머지 금지
</action_safety>

<default_follow_through_policy>
가장 합리적인 해석을 기본으로 삼고 계속 진행.
정보 부재가 잘못된 동작을 야기할 때만 멈춘다.
</default_follow_through_policy>
```

## 변수 치환 규칙

| 변수 | 출처 | 비고 |
|------|------|------|
| `GROUP_NAME` | 현재 MR을 생성할 그룹 | 예: `Group A` |
| `PLATFORM` | `glab` 또는 `gh` (자동 감지) | git remote URL 기반 |
| `MR_ENTRIES` | 해당 그룹의 task 브랜치 목록 | 아래 포맷 참조 |
| `MERGE_ORDER_NOTE` | 머지 순서 안내 문구 | 아래 참조 |

### MR_ENTRIES 포맷

```
- parallel/a1-db-schema → main (Task a1: DB 스키마)
- parallel/a2-api-types → main (Task a2: API 타입)
```

### MERGE_ORDER_NOTE 예시

- **Group A**: `"Group A에 속하며 같은 그룹 내 다른 MR과 순서 무관하게 머지 가능합니다."`
- **Group B+**: `"Group B에 속하며 Group A의 모든 MR이 main에 머지된 후 머지해야 합니다."`

## 플랫폼 감지

`git remote get-url origin`으로 판별:
- `gitlab` → `glab mr create --draft`
- `github` → `gh pr create --draft`

## Resume 시 호출

이 템플릿은 두 가지 시점에 호출된다:

1. **최초 실행 완료 후**: Group A의 MR 생성
2. **resume 시**: Claude가 이전 그룹 머지를 확인하고 `git rebase main`을 직접 실행한 뒤, 이 템플릿을 MR 생성 Codex에 전달한다.

Resume 시에는 Claude가 rebase를 선행하므로 diff가 해당 task 변경만 포함한다.
