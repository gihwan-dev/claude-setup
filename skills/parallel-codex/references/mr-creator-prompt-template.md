# MR Creator Prompt Template

모든 그룹의 실행·검증 완료 후, task별 Stacked MR을 생성하는 Codex 프롬프트 구조.
각 MR은 해당 task의 변경분만 diff에 표시되어 리뷰가 용이하다.
Claude가 변수를 치환한 뒤 Codex에 전달한다.

## Template

```xml
<task>
## 목적
파이프라인의 모든 작업을 선형 순서로 stack 브랜치에 쌓고, 각 task별 draft MR을 생성한다.
각 MR은 이전 task의 stack 브랜치를 타겟으로 하여 **해당 task의 변경분만** 표시한다.

## 플랫폼
${PLATFORM}

## Task 순서 (선형)
${TASK_ORDER}

## 1단계: Stack 브랜치 생성
아래 순서대로 stack 브랜치를 생성한다:

${STACK_STEPS}

각 단계 절차:
1. 이전 stack 브랜치(또는 main)에서 checkout: `git checkout -b <stack-branch> <base>`
2. task 브랜치를 머지: `git merge <parallel-branch> --no-edit`
3. 충돌 시: 검증 Codex가 이미 같은 조합을 검증했으므로 동일한 방식으로 해결

## 2단계: MR 생성
각 task에 대해 draft MR을 생성한다:

${MR_STEPS}

MR 공통 규칙:
- **draft**로 생성
- MR 제목: `<type>(<scope>): <task 설명>` (한국어, 70자 이내)
- MR body 포맷 (아래 구조대로 작성):

**MR Body:**
> ## Summary
> <task 설명 1-2문장>
>
> ## Stacked MR (${CURRENT_INDEX}/${TASK_COUNT})
> 이 MR은 Stacked MR 시리즈의 일부입니다. **순서대로** 리뷰·머지하세요.
>
> ${STACK_TABLE}
>
> ---
> *Generated by parallel-codex pipeline*

`STACK_TABLE`은 각 MR마다 동일한 구조이되, 현재 MR 행에 👈 표시를 한다.
이미 생성된 MR은 URL을 포함하고, 아직 미생성인 MR은 브랜치명만 표기한다.

## 완료 기준
- 모든 stack 브랜치 생성 완료
- 모든 draft MR 생성 완료 (${TASK_COUNT}개)
- 각 MR URL 수집
- PIPELINE.md의 MR Plan 상태 업데이트
</task>

<structured_output_contract>
완료 후 반드시:
1. 생성된 모든 MR 목록 (순서, 제목, Source→Target, URL)
2. PIPELINE.md의 MR Plan 테이블 업데이트 (각 행의 Status를 `created`로, URL 추가)
</structured_output_contract>

<action_safety>
- stack 브랜치 생성, MR 생성, PIPELINE.md 업데이트만 수행
- 코드 수정 금지
- 기존 parallel/* 브랜치 수정 금지
</action_safety>

<default_follow_through_policy>
가장 합리적인 해석을 기본으로 삼고 계속 진행.
정보 부재가 잘못된 동작을 야기할 때만 멈춘다.
</default_follow_through_policy>
```

## 변수 치환 규칙

| 변수 | 출처 | 비고 |
|------|------|------|
| `PLATFORM` | `glab` 또는 `gh` (자동 감지) | git remote URL 기반 |
| `TASK_ORDER` | PIPELINE.md Tasks+Groups에서 추출 | 그룹 순 → 그룹 내 ID 순 |
| `TASK_COUNT` | 총 task 수 | 예: `3` |
| `STACK_STEPS` | task 순서 기반 stack 브랜치 생성 절차 | 아래 포맷 참조 |
| `MR_STEPS` | task 순서 기반 MR 생성 절차 | 아래 포맷 참조 |
| `CURRENT_INDEX` | MR body 내 현재 MR 순번 | MR마다 다름 |
| `STACK_TABLE` | MR body에 삽입할 전체 스택 네비게이션 | 아래 포맷 참조 |

### TASK_ORDER 포맷

```
1. a1 — DB 스키마 (Group A)
2. a2 — API 타입 (Group A)
3. b1 — 엔드포인트 (Group B)
```

### STACK_STEPS 포맷

```
1. `git checkout -b stack/01-db-schema main` → `git merge parallel/a1-db-schema --no-edit`
2. `git checkout -b stack/02-api-types stack/01-db-schema` → `git merge parallel/a2-api-types --no-edit`
3. `git checkout -b stack/03-endpoint stack/02-api-types` → `git merge parallel/b1-endpoint --no-edit`
```

### MR_STEPS 포맷

```
1. MR: `stack/01-db-schema` → `main` — "feat(db): 유저 테이블 추가"
2. MR: `stack/02-api-types` → `stack/01-db-schema` — "feat(types): 유저 DTO 정의"
3. MR: `stack/03-endpoint` → `stack/02-api-types` — "feat(api): 유저 CRUD API"
```

### STACK_TABLE 포맷

MR body에 삽입되는 스택 네비게이션 테이블.
각 MR마다 `CURRENT_INDEX`가 달라 👈 위치가 바뀐다.
이미 생성된 MR은 URL 링크를, 미생성 MR은 브랜치명을 표기한다.

예시 (MR-2의 body):

```
| # | Task | Source → Target | Link |
|---|------|-----------------|------|
| 1 | a1: DB 스키마 | `stack/01-db-schema` → `main` | !123 |
| **2** | **a2: API 타입** | **`stack/02-api-types` → `stack/01-db-schema`** | **👈 현재** |
| 3 | b1: 엔드포인트 | `stack/03-endpoint` → `stack/02-api-types` | (다음) |
```

## 플랫폼 감지

`git remote get-url origin`으로 판별:
- `gitlab` → `glab mr create --draft`
- `github` → `gh pr create --draft`
