# MR Creator Prompt Template

모든 그룹의 실행·검증 완료 후, 마지막 verify 브랜치에서 main으로 단일 MR을 생성하는 Codex 프롬프트 구조.
Claude가 변수를 치환한 뒤 Codex에 전달한다.

## Template

```xml
<task>
## 목적
파이프라인의 모든 작업이 통합·검증된 `${VERIFY_BRANCH}` 브랜치에서 `main`으로 단일 draft MR을 생성한다.

## 플랫폼
${PLATFORM}

## MR 소스 브랜치
${VERIFY_BRANCH} → main

## MR 생성 규칙
- **draft**로 생성
- MR 제목: `<type>(<scope>): <설명>` (한국어, 70자 이내). 전체 파이프라인의 목적을 요약.
- MR body 포맷 (아래 구조대로 작성):

**MR Body:**
> ## Summary
> ${PIPELINE_SUMMARY}
>
> ## Included Tasks
> ${TASK_LIST}
>
> ## Groups
> ${GROUP_SUMMARY}
>
> ---
> *Generated by parallel-codex pipeline*

## 완료 기준
- `${VERIFY_BRANCH} → main` draft MR 1개 생성
- MR URL 수집
- PIPELINE.md의 MR Plan 상태 업데이트
</task>

<structured_output_contract>
완료 후 반드시:
1. 생성된 MR (제목, URL)
2. PIPELINE.md의 MR Plan 테이블 업데이트 (Status를 `created`로, URL 추가)
</structured_output_contract>

<action_safety>
- MR 생성과 PIPELINE.md 업데이트만 수행
- 코드 수정 금지
- 브랜치 머지 금지
</action_safety>

<default_follow_through_policy>
가장 합리적인 해석을 기본으로 삼고 계속 진행.
정보 부재가 잘못된 동작을 야기할 때만 멈춘다.
</default_follow_through_policy>
```

## 변수 치환 규칙

| 변수 | 출처 | 비고 |
|------|------|------|
| `VERIFY_BRANCH` | 마지막 그룹의 Verify Branch | 예: 그룹이 A, B면 `temp/verify-b` |
| `PLATFORM` | `glab` 또는 `gh` (자동 감지) | git remote URL 기반 |
| `PIPELINE_SUMMARY` | 전체 파이프라인 목적 1-2문장 요약 | Claude가 작성 |
| `TASK_LIST` | 전체 task 목록 (ID, 이름, 그룹) | 아래 포맷 참조 |
| `GROUP_SUMMARY` | 그룹별 실행 결과 요약 | 아래 포맷 참조 |

### TASK_LIST 포맷

```
- **a1** (Group A): DB 스키마 — 유저 테이블 추가
- **a2** (Group A): API 타입 — 유저 DTO 정의
- **b1** (Group B): 엔드포인트 — 유저 CRUD API
```

### GROUP_SUMMARY 포맷

```
- **Group A**: a1, a2 — 병렬 실행 완료, 검증 통과
- **Group B**: b1 — 병렬 실행 완료, 검증 통과
```

## 플랫폼 감지

`git remote get-url origin`으로 판별:
- `gitlab` → `glab mr create --draft`
- `github` → `gh pr create --draft`
